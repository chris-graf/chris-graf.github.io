Why Work In Branches?

If you're not a squirrel

2015 July 11

The master is supposed to be the authoritative, curated version of the project. Commits should only be pushed to master once they've been reviewed, tested, and certified. You want to ensure not only that your commits are functional themselves, but also that they area free of negative side effects. You do this by working in branches, where you can bork up as much as you want without affecting the master.

This doesn't matter so much when you're only building out some personal side project. When you're working on live code in a professional context with a team of developers, though, you want to be absolutely sure you're not breaking existing features, exposing vulnerabilities, or otherwise introducing chaos before committing to master.
