No Method to the Madness: JavaScript Functions for the Inquiring Rubyist

2015 August 23

JavaScript's take on the object-oriented paradigm may not seem as clean or elegant as Ruby's at first look, but in some ways, JavaScript is a more object-y language than Ruby.


First, let's take a look at how you might, as a Rubyist would say, construct a class in the two different languages. In Ruby:

class CoolClass def initialize(some_number)
@fun_level = some_number
end def pump_it_up @fun_level += 1 end end my_cool_class = CoolClass.new(99) my_cool_class.fun_level => 99 my_cool_class.pump_it_up my_cool_class.fun_level => 100 And in JavaScript: 
              

                var CoolClass = function(someNumber) {
                  this.funLevel = someNumber;
                  this.pumpItUp = function() {
                    this.funLevel++;
                  }
                }

               var myCoolClass = new CoolClass(99);
               myCoolClass.funLevel;
               // 99
               myCoolClass.pumpItUp();
               myCoolClass.funLevel;
               // 100


(These examples don't necessarily represent the idealized form of either language - the Ruby example would almost certainly include at least an attr_reader to wrap the @fun_level instance variable, for examle - but for the purpose of demonstrating equivalence or lack thereof, they'll do.)


The main difference, perhaps, is how clumpy the JavaScript implementation is compared to the Ruby one. In Ruby's CoolClass, we've got the intialize method separated from pump_it_up. Each method contains some expression involving variables, and the methods in turn are contained by the class. There's a clear delineation between the methods, and the objects they operate upon. In JavaScript, we don't have this obvious separation of method and object. Instead, our "class" is a function being assigned to a variable (this would be less obvious if we defined this function with a declaration instead of an expression, but in this context the result is the same). This function constructs an object with two properties, one of which is another function. And in JavaScript, it turns out, functions, properties, and variables are all just various grades of object. At the fundamental level, our JavaScript class is simply creating and connecting objects to one another, with no major distinctions.


It is this total reliance on objects that allows one to argue that JavaScript is more purely object-oriented than Ruby. That's not a value judgement, and there are counterarguments aplenty (nevermind exceptions to every rule we mentioned above), but these example nonetheless illustrate one of the fundamental differences in how these two languages approach the OO paradigm.
